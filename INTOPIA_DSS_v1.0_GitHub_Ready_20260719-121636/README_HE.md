# EMBA TAU Simulation AI Decision OS — v1.0

פלטפורמת ענן תומכת החלטה לסימולציית INTOPIA. המערכת מתחילה את עבודת התכנון בתחילת Q4, לאחר קליטה נפרדת של תוצאות Q1, Q2 ו־Q3, ומסתכלת תמיד קדימה עד Q9.

## מה חדש ב־v1.0

- Rulebook מאושר ומבוסס מקורות, עם מזהה, תחום, גרסה, עמוד, רמת ודאות ואופן אכיפה לכל חוק.
- מרכז חוקים לחיפוש בשפה רגילה ולבדיקת השאלה “האם פעולה זו מותרת?”.
- אכיפה דטרמיניסטית לאחר חילוץ דוח, בזמן סימולציה ולפני יצירת חבילת החלטות.
- חבילת החלטות רבעונית הנחסמת אם קיימת הפרת חוק, חריגת תקציב, ירידה מתחת לרצפת המזומן או שגיאת תזמון.
- Decision Agent הפועל כצוות אנליסטים וירטואלי, משתמש בכלי Rulebook וסימולציה ושומר Audit של הרצותיו.
- כל מסמך קריא שנשמר באפליקציה מפורק לקטעי מקור הניתנים לחיפוש על ידי ה־AI; מסמך Setup חדש עובר ניתוח מועמדי חוקים כאשר ה־Agent מחובר.
- חוק חדש או סתירה ממסמך מועלה אינם משנים אוטומטית את ה־Rulebook: הם נכנסים לתור אישור אנושי במרכז החוקים.
- ישויות ענן חדשות לגרסאות חוקים, מקורות, סתירות, ראיות, תחזיות, אופטימיזציות וחבילות החלטה.
- גיבוי ושחזור כוללים גם את גרסת ה־Rulebook ואת כל ישויות v1.0.
- Q1–Q3 האמיתיים נבדקו: כל קובץ זוהה ברבעון הנכון, עם 4 אזורים, 12 שורות פעילות ומחקרי השוק שבתוכו, ללא שגיאות חילוץ.

ההפעלה הראשונה של v1.0 על מסד קיים דורשת הרצה חד־פעמית של `supabase/migration_v1.0.sql`. מדריך מדויק נמצא ב־`DEPLOY_v1.0_STEP_BY_STEP_HE.md`.

## יכולות חדר ההחלטות

- שישה טאבים מרכזיים בלבד: חדר החלטות, צ׳אט AI, מצב פיננסי, תובנות ומגמות, לוג החלטות והגדרות.
- תוצאות מחקרי שוק מוצגות עם ערכי המקור המדויקים, מספר MR, רבעון, מקור, ודאות והמלצה מעשית.
- זיהוי מגמות מצטבר בפיננסים, תמחור, מכירות, מלאי, מתחרים ומחקרי שוק משולבים.
- העלאת קבצים ואישור החילוץ ישירות מתוך צ׳אט ה־AI.
- יומן החלטות עם המלצת המערכת, תוצאה צפויה, ביצוע בפועל ולמידה בין רבעונים.
- תצוגה פיננסית לפי רבעון או טווח מצטבר, מאוחד ולפי מדינה.

### יכולות התשתית שנשמרו מ־v0.6

- ממשק הנהלה פיננסי חדש ותפריט המבורגר סגור כברירת מחדל.
- מפענח ייעודי לפלטי INTOPIA האמיתיים: Balance Sheet, Income Statement, Management Info, Currency, MR17/28, MR74 וכל גיליון MR נוסף שהוזמן.
- זיהוי רבעון אוטומטי משם הקובץ; Q1, Q2 ו־Q3 נשמרים בנפרד ומכיילים את תמונת הפתיחה ל־Q4.
- קליטה מאוחדת ולפי USA, Europe, Brazil ו־Liechtenstein, כולל שערי המטבע של הרבעון.
- מחקרי Q2 בדוגמת הכיול מזוהים מתוך אותו workbook, לרבות טווחי MR41/MR61 ומיפוי MR40; אין צורך להעלות אותם שוב.
- דוח רבעוני ודוח מצטבר לפי כל הרבעונים שאושרו.
- מצב כספי מאוחד ולפי מדינה, כולל מזומן, חוב, הון חוזר, מלאי והתחייבויות.
- אומדן ניהולי שקוף: 50% ביצועי עבר ו־50% פוטנציאל עתידי. זה אינו הציון הרשמי של המשחק.
- תחזית Rolling ל־Q9, זיהוי מגמות, המלצות תמחור, קיבולת, מו״פ ומחקרים רלוונטיים.
- מעבדת תרחישים עם כל 19 טופסי ההחלטה שנמצאו בחומרי המשחק: A1-1–A4, H1-1–H6 ו־W1–W3.
- בדיקות אוטומטיות לתקציב, רצפת מזומן, קיבולת, עד שלושה מפעלים, תאימות X–Y ותזמון פעולות.
- Decision Agent אופציונלי של OpenAI. הוא קורא רק נתונים מאושרים דרך כלי קריאה ואינו משנה נתונים.
- שמירת נתונים ב־Supabase PostgreSQL וקבצים ב־Supabase Storage; אין SQLite ואין הסתמכות על הדיסק הזמני של Render.
- גיבוי ושחזור ZIP מלאים, כולל נתונים, קבצים, imports, מחקרים, תרחישים ושיחות Agent.

## מסלול העבודה

1. העלו מסמך אסטרטגיה ויעדי Q9.
2. בחרו **זיהוי רבעון אוטומטי** והעלו בנפרד את פלט Q1, את פלט Q2 ואת פלט Q3.
3. עברו על תצוגת החילוץ ולחצו **אישור קליטה** רק לאחר בדיקה.
4. עבור מסמך חוקים חדש, עברו למרכז החוקים ואשרו, דחו או השאירו לבדיקה כל מועמד. אישור מייעד אותו לגרסת Rulebook הבאה ואינו משנה את הגרסה הפעילה מיד.
5. השלימו רק נתונים שהמערכת לא הצליחה לחלץ, בעיקר מידע איכותני.
6. פתחו את **חדר ההחלטות Q4**: מצב כספי, ציון פנימי, תחזית Q9 והמלצות.
7. העבירו פעולות אפשריות למעבדת התרחישים ובדקו אותן מול התקציב ורצפת המזומן.
8. בכל סוף רבעון העלו את הפלט החדש, אשרו אותו והורידו גיבוי.

## משתני הסביבה שחייבים להזין ב־Render

| משתנה | מה מזינים |
|---|---|
| `SUPABASE_URL` | כתובת Project URL של Supabase |
| `SUPABASE_SECRET_KEY` | Secret key או `service_role` של השרת בלבד |
| `APP_ACCESS_PASSWORD` | סיסמת צוות חדשה וחזקה |

ה־Blueprint כבר מגדיר `INTOPIA_BACKEND=supabase`, את ה־bucket הפרטי `intopia-files`, הגנת סיסמה ומגבלות קבצים.

## הפעלת Decision Agent

הוסיפו ב־Render רק אם רוצים להפעיל את הצ׳אט:

```text
OPENAI_AGENT_ENABLED=true
OPENAI_API_KEY=<מפתח שרת של פרויקט OpenAI>
OPENAI_MODEL=gpt-5.6-sol
OPENAI_MAX_OUTPUT_TOKENS=2400
```

ה־Blueprint כבר מגדיר `OPENAI_AGENT_ENABLED=true`, את `OPENAI_MODEL=gpt-5.6-sol` ואת מגבלת הפלט; בפועל נדרש להזין בעצמכם רק את `OPENAI_API_KEY`. המפתח נשמר ב־Render בלבד. אין לשים אותו ב־GitHub, בקובץ JavaScript, בצילום מסך או בצ׳אט. שימוש ב־OpenAI API כרוך בחשבון API ובמכסה נפרדים מ־ChatGPT. המסך `/api/agent/status` מציג בדיוק איזה משתנה חסר בלי להציג את ערכו.

## מגבלות חילוץ קבצים

- CSV ו־Excel עם כותרות ברורות: חילוץ אוטומטי + מסך אישור.
- PDF המכיל טקסט: חילוץ מדדים בסיסיים + מסך אישור.
- PDF סרוק ותמונה: הקובץ נשמר, אך מסומן כדרוש OCR/בדיקה. המערכת אינה ממציאה מספרים שלא זוהו.
- פלט INTOPIA במבנה שנבדק על Q1–Q3: מפענח מדויק ובדיקות התאמה.
- שינוי עתידי במבנה גיליונות המשחק יסומן לבדיקה ולא יומר למספרים מומצאים.

## בדיקות

```text
python -m pip install -r requirements-dev.txt
python -m pytest -q
```

בדיקת Supabase אמיתית מופעלת רק עם `RUN_SUPABASE_INTEGRATION=1`. לעולם אין לבצע commit של `.env`, מפתחות, דוחות אמיתיים או קובצי גיבוי.
